# PLAN: ターン制・肩越しTPS作戦ローグライト MVP計画

## Goal

肩越しTPS視点、ターン制、ノード移動、壊れる銃、有限弾薬、危険な支援要請、主目標達成後の抽出フェーズがゲームとして成立するかを検証する。

MVPの目的は、コンテンツ量ではなく以下の問いに答えることである。

- 肩越しTPS視点でターン制は気持ちよいか
- ノード移動は3D空間と相性がよいか
- 射線、遮蔽、高低差が判断材料になるか
- 武器破損はストレスではなく判断になるか
- リロードとタクティカルリロードに意味があるか
- 支援要請の遅延、巻き込み、破壊は面白いか
- 主目標達成後の抽出フェーズは緊張感を生むか
- 失敗しても一部報酬が残る設計は納得感があるか

## Prototype Design

### Combat Arena

MVPでは1つの小型3Dアリーナを作る。

要件:

- 10〜20個程度のノード
- 高さ2〜3段階
- 遮蔽ノード
- 高所ノード
- 屋内/屋外の差
- 破壊可能な橋または床
- 抽出地点
- 主目標地点
- 支援要請が有効な開けた地点

### Node Types

最低限のノード属性:

- Normal
- HalfCover
- FullCover
- HighGround
- LowGround
- Ladder
- Bridge
- Destroyable
- Indoor
- Extraction
- Objective

### Turn Actions

すべての主行動は1ターンを消費する。

- Move
- Shoot
- Reload
- CallSupport
- UseItem
- Combine
- Disassemble
- Wait

情報確認はターンを消費しない。

- AimPreview
- LineOfSightPreview
- InventoryView
- SupportPreview
- DamagePreview

## Movement

### Rule

プレイヤーは1ターンで隣接ノードへ1つ移動できる。

### Requirements

- 隣接ノード表示
- 移動先の遮蔽状態表示
- 高低差表示
- 移動後の射線プレビュー
- 支援範囲内かどうかの警告
- 破壊済みノードは通行不可

## Shooting

### Rule

射撃は射線判定を基本とする。

- 射線が通るなら命中確定寄り
- 遮蔽、距離、高低差で効果量を変える
- 命中率で外すより、ダメージや追加効果の減衰で調整する

### MVP Weapons

MVPでは以下の武器のみ実装する。

#### Rescue Handgun

- 固定装備
- 弾無限
- 低威力
- 低射程
- 壊れない
- 特殊効果なし
- 詰み防止用

#### Assault Rifle

- メイン武器
- 3点バースト
- 軽弾を消費
- 中距離向き
- 標準的な耐久

#### Shotgun

- セカンダリまたはメイン武器
- 近距離高威力
- ノックバックあり
- 散弾を消費
- 落下や位置取りと相性がよい

#### Launcher

- パワー武器
- 範囲攻撃
- 地形破壊あり
- 弾薬少なめ
- 味方巻き込みあり
- 耐久消耗大

### Ammo Types

MVPでは以下に絞る。

- LightAmmo
- ShellAmmo
- HeavyAmmo

必要ならPowerAmmoを追加する。

## Reload

### Rule

リロードは1ターンを消費する。

- リロード自体では武器耐久は減らない
- マガジンが空なら通常リロード
- マガジンに弾が残っている場合はタクティカルリロード
- タクティカルリロードではマガジン内残弾を破棄する

### Required UI

- 現在マガジン弾数
- 所持弾薬
- リロード後の弾数
- タクティカルリロード時に破棄される弾数
- リロード後に敵に攻撃される可能性の警告

## Weapon Durability

### Rule

武器耐久は発砲で減少する。

- 発砲数または武器ごとの消耗係数で現在耐久を減らす
- 現在耐久が0になると破損
- 破損武器は射撃不可
- 破損武器は分解またはニコイチ素材にできる

### MVP Durability Model

MVPではシンプルに以下で始める。

- Rescue Handgun: 耐久減少なし
- Assault Rifle: 射撃行動ごとに耐久 -2
- Shotgun: 射撃行動ごとに耐久 -1
- Launcher: 射撃行動ごとに耐久 -2

発砲ごとの減少が過酷すぎる場合に備え、MVPでは「射撃行動ごとの耐久減少」を暫定採用してもよい。  
最終判断はプレイテストで行う。

## Combine

### Rule

ニコイチは1ターンを消費する。

効果:

- ベース武器の現在耐久を回復
- ベース武器の最大耐久 -1
- ベース武器の摩耗 +1
- 素材武器の特性を1つ継承
- 素材武器は消滅

### MVP Simplification

MVPでは特性継承を簡略化してよい。

例:

- DamageBonus
- KnockbackBonus
- RangeBonus
- DurabilityRecoveryOnly

まずは「現在耐久回復 + 最大耐久低下 + 摩耗増加」だけでも検証可能。

## Disassemble

### Rule

分解は1ターンを消費する。

分解結果:

- 弾薬
- 修理素材
- 支援ビーコン
- ニコイチ素材

MVPでは分解結果を固定でもよい。

例:

- AR分解: LightAmmo + Scrap
- Shotgun分解: ShellAmmo + Scrap
- Launcher分解: HeavyAmmo + Scrap

## Support Call

### Rule

支援要請は1ターンを消費する。

- 適用ポイントを指定する
- 1〜5ターン後に発動する
- 敵味方を区別しない
- 地形を破壊する
- 予測範囲を表示する

### MVP Supports

#### Precision Strike

- 遅延: 2ターン
- 範囲: 小
- 高ダメージ
- 地形破壊小
- 味方巻き込みあり

#### Supply Pod

- 遅延: 2ターン
- 着弾地点にダメージ
- 弾薬と素材を補給
- 地形破壊小
- 敵にも当たる

#### Orbital Barrage

- 遅延: 4ターン
- 範囲: 大
- 高威力
- 地形破壊大
- 味方巻き込み大
- MVPでは1回限定でもよい

### Required UI

- 発動までの残りターン
- 爆心地
- ダメージ範囲
- 地形破壊範囲
- 巻き込まれる対象
- 抽出路への影響
- 警戒レベル上昇の予告

## Enemies

### MVP Enemy Types

#### Melee Enemy

- 近接攻撃
- プレイヤーへ接近
- 支援範囲から逃げにくい

#### Rifle Enemy

- 遮蔽から射撃
- 射線が通る位置へ移動
- 高所を優先する

#### Heavy Enemy

- 高耐久
- 低速
- パワー武器や支援を使いたくなる相手

#### Jammer Enemy

- 支援要請の遅延増加
- 一定範囲で支援不可
- 優先撃破対象

MVPでは最初の3種類だけでもよい。  
支援要請の面白さを検証するならJammer Enemyを早めに入れる。

## Mission Flow

### MVP Mission

1. 初期支給を選ぶ
2. アリーナに降下
3. 敵を避ける、倒す、または支援で処理する
4. 主目標に到達
5. 主目標を1ターンかけて達成
6. 警戒レベル上昇
7. 抽出地点が有効化
8. 抽出地点へ移動
9. 抽出要請
10. 3ターン耐える
11. 成功または死亡

### Objective Examples

MVPでは以下から1つ。

- データ回収
- 通信塔破壊
- 敵ジャマー停止
- 補給物資回収
- ビーコン設置

## Extraction

### Rule

主目標達成後、抽出地点へ移動する。

抽出地点で抽出要請を行うと、抽出カウントが始まる。

MVP標準:

- 抽出待機: 3ターン
- 敵増援あり
- 支援要請は可能
- 抽出地点が破壊された場合は失敗または別地点へ移動

### Extraction Tension

抽出中に発生させたい判断:

- 残弾を温存するか
- ハンドガンでしのぐか
- 支援で敵を止めるか
- 支援で抽出地点を壊すリスクを取るか
- 武器をニコイチする1ターンを使うか
- 待機して支援着弾を待つか

## Rewards

### Success

抽出成功時に得るもの:

- BP断片
- 物資
- 戦闘ログ
- 支援候補
- 一時セーフティゾーン候補

MVPでは以下だけでよい。

- BP断片 +3
- 戦闘ログ +1
- 物資 +1

### Failure After Objective

主目標達成後に死亡した場合:

- BP断片 +1
- 戦闘ログ +1
- 物資なし

### Failure Before Objective

主目標未達で死亡した場合:

- 戦闘ログ +1
- BP断片なしまたは少量

## Blueprint

### MVP Rule

BPは初期支給候補の解放に使う。

MVPでは以下のようにする。

- BP断片を一定数集める
- 初期支給候補が1つ増える
- 支給品は低耐久、低弾薬、低素材価値

### MVP Initial Loadout Candidates

#### Standard Kit

- Assault Rifle
- LightAmmo
- Supply Beacon x1

#### Breacher Kit

- Shotgun
- ShellAmmo
- Smoke or Precision Strike x1

#### Heavy Kit

- Weak Launcher
- HeavyAmmo少量
- 低耐久

#### Scout Kit

- Rescue Handgun中心
- 支援ビーコン多め
- 敵位置情報少し

## Safety Zone

MVPでは本実装しない。

ただし報酬として以下の仮データを表示する。

- Temporary Supply Point unlocked for 2 operations
- Temporary Extraction Point unlocked for 1 operation
- Communication Relay unlocked for 2 operations

実際の効果は後続フェーズで実装する。

## Implementation Phases

### Phase 0: Paper Prototype

目的:

- ターン制ルールの確認
- 射撃、リロード、支援遅延の判断確認
- 抽出フェーズの面白さ確認

成果物:

- 10〜15ノードの紙マップ
- 武器カード
- 支援カード
- 敵カード
- 1ミッションの手動プレイ

完了条件:

- 1ミッションを手動で通しプレイできる
- 抽出フェーズで判断が発生する
- 支援要請が強いが危険に感じる

### Phase 1: UE5 Greybox

目的:

- 肩越しTPS視点とノード移動の相性確認

実装:

- TPSカメラ
- ノード選択移動
- 1アリーナ
- プレイヤー
- 敵1種
- 射線プレビュー
- 射撃
- ターン進行

完了条件:

- 移動、射撃、敵ターンが成立する
- 肩越し視点でノード選択が破綻しない
- 射線と遮蔽が理解できる

### Phase 2: Weapon Resource Prototype

目的:

- 弾薬、マガジン、リロード、耐久、破損の検証

実装:

- 武器3種
- 弾薬2〜3種
- マガジン
- リロード
- タクティカルリロード
- 耐久減少
- 破損
- 救済ハンドガン

完了条件:

- リロード判断が発生する
- 武器破損が理不尽すぎない
- 救済ハンドガンが詰み防止として機能する

### Phase 3: Support Prototype

目的:

- 支援要請の遅延、巻き込み、破壊の検証

実装:

- 支援2〜3種
- 範囲予告
- 遅延発動
- 敵味方巻き込み
- 地形破壊
- 支援カウントUI

完了条件:

- 支援要請が強いが雑に使えない
- 自爆や地形破壊が納得できる
- 支援待ちのターンが面白い

### Phase 4: Extraction Prototype

目的:

- 主目標達成後の帰還フェーズの検証

実装:

- 主目標
- 警戒レベル上昇
- 抽出地点
- 抽出待機3ターン
- 増援
- 成功/失敗報酬

完了条件:

- 目標達成後も緊張感が残る
- 帰還ルート選択に意味がある
- 支援で撤退路を壊すリスクが機能する

### Phase 5: Meta Prototype

目的:

- 失敗しても少し報酬が残る構造の検証

実装:

- BP断片
- 戦闘ログ
- 初期支給候補解放
- 簡易報酬画面
- 失敗時報酬

完了条件:

- 死亡しても完全無駄に感じない
- 抽出成功の価値が残る
- 初期支給候補がマンネリを減らす

## Key Balancing Questions

### Turn Rules

- 1ターン1行動は重すぎないか
- 移動1ノードが遅すぎないか
- 待機1ターンは有効な判断になるか

### Weapons

- 武器は壊れやすすぎないか
- 支給品は強すぎないか
- 救済ハンドガンは弱すぎて退屈ではないか
- タクティカルリロードの弾破棄は十分なデメリットか
- 弾薬種は多すぎないか

### Support

- 支援要請は強すぎないか
- 遅延は長すぎないか
- 地形破壊は面白さにつながるか
- 巻き込み事故は理不尽ではなく納得感があるか

### Extraction

- 抽出待機3ターンは適切か
- 主目標達成後の増援はストレスになりすぎないか
- 帰還ルートに往路の判断が反映されるか

### Meta

- 失敗時報酬は多すぎないか
- 成功報酬との差は十分か
- BPで初期支給候補が増えることはマンネリ解消になるか
- BPが最適解固定を生まないか
